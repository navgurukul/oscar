Oscar AI Backend Split

Why Scribble runs on Amplify and Stream runs on Supabase — and whether to consolidate.

Design memo · 2026-06-22 · data: 1,163 perf.jsonl records (May 27 – Jun 22), 200 with server-side timing split

STATUS UPDATE · 2026-06-30 — recommendation shipped, with a caveat.

Plan A executed: dictation cleanup cut over to the Amplify web route /api/ai/dictation-cleanup (commit 1f7ee1a, default-on since v0.11.0). Warm-path latency improved as predicted (cleanup roundtrip p50 ~1413ms → ~664ms).

The flagged cold-start risk materialized: on-device perf.jsonl (Jun 30) showed ~17% of dictations degraded by Amplify Lambda cold-starts — some past the 7s deadline, pasting raw unpolished text. Mitigation shipped: the warmUp() prewarm now fires at record-start (was processing-start) for far more lead time. Measuring before heavier levers.

Decisions: keep the ai-process Edge Function (still serves Scribble transforms + meeting fallback — only transcribe_cleanup moved); keep the VITE_STREAM_CLEANUP_VIA_WEB flag as the rollback lever until cold-start is proven stable. Live status + remaining levers (scheduled keep-warm ping, deadline tuning, edge cleanup-path retirement) tracked in ROADMAP.md → “Dictation Latency & AI-Backend Consolidation”.

TL;DR

The split is half deliberate, half accreted. Scribble→Amplify is a sound, intentional decision (single prompt + rate-limit + quota source of truth, shared web/desktop). Stream→Supabase-edge is the older path that grew on its own — it carries the dual-key footgun (the bug you just hit) and a duplicated Mercury client.

Recommendation: consolidate onto the web routes (Amplify), retire the edge AI path. Measured data shows it's latency-neutral-to-better when warm; the one real risk is Lambda cold-starts, which you already have prewarm tooling for.

Two independent copies of the same secret. Set one, forget the other → half the AI silently falls back. That's the operational footgun baked into the split — and exactly the title/format bug.

What the latency data actually says

The memory note claimed "~69% is the Supabase edge double-hop." Measured against 200 instrumented records, that framing is imprecise. The cleanup roundtrip decomposes as:

Mercury 581ms · 43% edge-internal ~530ms wire 289ms
Mercury inference (real model work) Edge function non-Mercury work (org-context Supabase lookups, prompt assembly) Client↔edge wire (cleanup-platform-net)

p50 values. The overhead isn't pure network — the bigger non-Mercury chunk is the edge function doing org-context fallback lookups back into Supabase REST. That work would cost the same on Amplify unless trimmed — and the web title route already dropped it (saved ~7 Supabase round-trips).

Both resolve to the same local PoP (CCU). Warm Amplify is ~2× faster on the wire than the edge — the edge is not buying the latency advantage its design implies. The catch is Amplify's cold-start (~2.8s); the edge degrades far more gracefully when idle.

Two clean end-states

A · Everything on Amplify recommended

Retire ai-process; stream cleanup calls a web route like the Scribble paths.

Gains

  • One key, one Mercury client, one prompt source — kills the dual-key bug & the duplication
  • Warm wire is faster (160 vs 300 ms)
  • Scribble decision (the keeper) becomes the only pattern

Costs / risks

  • Lambda cold-start ~2.8s — mitigated by existing prewarm + provisioned concurrency
  • Lose edge global distribution (unrealized today anyway)
  • Must port org-context logic (or drop it for stream, as title did)

B · Everything on the edge

Move Scribble format/title onto ai-process too.

Gains

  • One key/codebase; keeps edge distribution & graceful idle

Costs / risks

  • Reverses the deliberate b8fdb6d decision
  • Web Scribble streaming gets awkward through edge
  • Web app's AI now depends on Supabase CLI deploys — decoupled from Amplify releases, a coordination wrinkle
  • Slower warm wire than Amplify

Recommendation

Go with A — consolidate onto Amplify web routes, retire the edge ai-process AI path. The Scribble→web decision is the keeper; make it the single pattern. It's latency-neutral-to-better warm, and removes both the dual-key footgun and the duplicated Mercury client.

Sequence, lowest-risk first:

  • 1 · Stop the bleeding now — set MERCURY_API_KEY in Amplify (unblocks titles + format today). Independent of any consolidation.
  • 2 · Add a stream-cleanup web route mirroring ai-process's cleanup prompt; point the desktop pill at it behind a flag.
  • 3 · A/B the latency on real devices via perf.jsonl (warm-path win vs cold-start tail) before flipping the default.
  • 4 · Decide org-context for stream — port it or drop it (title already dropped it cleanly).
  • 5 · Retire ai-process's Mercury path once the web route is proven; one key, one client.

Caveats on this analysis

  • Scribble flow is uninstrumented — all 1,163 records are stream-dictation. The Amplify-side latency numbers are raw wire probes, not real Scribble-pipeline timings. Step 3 would close that gap.
  • Wire probes are from one machine / one region (CCU PoP). A user far from the Amplify region would see a worse cold-start story; the edge's distribution matters more for them.
  • Cold-start risk is real but bounded — depends on Amplify keeping the AI Lambda warm under actual traffic; the existing ai-process prewarm pattern is portable.
